## 1. Goal

Allow a tenant OWNER to create or resume the tenant's one Stripe connected-account
onboarding flow from Organisation payment settings, allow OWNER and ADMIN to view safe
readiness, and allow only an OWNER to enable or disable future online checkout locally.

A6-B uses Stripe-hosted onboarding and the retained A6-A schema. Tenants enter no API key,
webhook secret, KYC, bank or card data into IsoStack.

## 2. Entry Baseline And Boundary

- application baseline is committed A6-A `513cf3a`;
- the complete migration inventory is 140 and A6-B adds no migration;
- Stripe SDK remains `22.2.0`, pinned to `2026-05-27.dahlia`;
- `STRIPE_SECRET_KEY` remains the platform deployment secret;
- `Organization.stripeCustomerId` and `/api/webhooks/stripe` remain subscription-only;
- A6-A connection, onboarding-attempt and audit/idempotency evidence is authoritative.

No real Stripe account or Account Link may be created in automated tests. Tests use an
injected fake provider adapter and only the disposable `TEST_DATABASE_URL` for persistence.

## 3. Stable Provider Contract

Create Accounts v1 connected accounts using the pinned stable controller shape:

```text
controller.fees.payer = account
controller.losses.payments = stripe
controller.requirement_collection = stripe
controller.stripe_dashboard.type = full
capabilities.card_payments.requested = true
```

Do not adopt Accounts v2 or deprecated account `type`/OAuth onboarding.

Use only `account_onboarding` Account Links with `collection_options.fields =
eventually_due` and `future_requirements = include`. `account_update` is not valid for a
full-Dashboard account whose requirements collector is Stripe.

The full Dashboard is opened through Stripe's normal Dashboard URL; do not create Express
login links. Stable Stripe does not expose a safe platform-side disconnect/delete operation
for this full-Dashboard, Stripe-loss-liability contract. Therefore A6-B:

- always permits immediate local checkout disablement;
- never calls `accounts.del`, rejects the account or invents OAuth disconnection;
- explains that account management/closure occurs in Stripe Dashboard;
- leaves verified provider disconnection/deauthorization synchronization to A6-D.

## 4. Authorization And Tenant Context

- OWNER and ADMIN may read the tenant's safe status;
- only OWNER may create/resume onboarding or change local checkout enablement;
- use effective impersonation identity only when an existing audited support path has
  established it; never let a platform administrator target an arbitrary tenant ID;
- every query and mutation derives tenant and actor from protected context;
- mutation services re-read the exact User `(organizationId,id)` role/status rather than
  trusting a stale client value;
- cross-tenant connection, attempt or actor IDs are rejected.

## 5. Provider Adapter And Safe Snapshot

Add a narrow injectable Stripe Connect adapter for:

- account create;
- account retrieve;
- Account Link create.

It receives explicit idempotency keys for every Stripe POST. Production uses the existing
lazy Stripe client; tests use a deterministic fake. Neither adapter nor service logs an
Account Link URL, state token, secret, full Account response or provider error body.

Normalize only safe Account evidence into A6-A:

- exact `acct_...` identity and server-derived credential mode;
- card capability, charges, payouts and details readiness;
- fixed controller contract;
- country/default currency;
- bounded disabled reason and requirement counts;
- SHA-256 fingerprint of sorted requirement field names/status, never the field values;
- sync/provider timestamps and pinned API version.

Reject a controller/account/mode mismatch without overwriting the prior accepted snapshot.
READY requires the existing A6-A conservative shape. Sync can force local checkout off
when readiness is lost; it cannot silently turn checkout on.

## 6. Idempotent Onboarding Lifecycle

### 6.1 Start or resume

OWNER supplies a UUID-shaped operation key generated by the settings client. The service:

1. claims A4 idempotency under tenant scope `STRIPE_CONNECT.ONBOARD`;
2. serializes per tenant using a transaction advisory lock;
3. reuses the current retained connection or creates one Stripe account with a stable
   Stripe idempotency key derived from tenant and monotonic connection generation;
4. validates and persists the provider account snapshot without using
   `Organization.stripeCustomerId`;
5. terminates any prior open attempt before creating a new attempt;
6. generates a cryptographically random state token, stores only its SHA-256 hash and a
   short expiry, and binds it to tenant, connection and initiating OWNER;
7. generates allowlisted server-owned return/refresh URLs containing the opaque state;
8. creates one single-use Account Link and returns its URL only to that authenticated
   request;
9. emits A4 Commerce and core tenant audit evidence without secrets and completes the
   idempotency record.

If Stripe succeeds but local persistence fails, retrying the same operation uses the same
Stripe idempotency key. A conflict/in-progress response creates no second account.

### 6.2 Return

The protected return route requires the same tenant OWNER who initiated the attempt. It
hashes the supplied state, locks the attempt, rejects invalid/expired/terminal tokens,
consumes it once, retrieves the canonical Account, synchronizes the safe snapshot and
redirects to `/settings/payments` without the state token. Leaving hosted onboarding does
not imply READY.

### 6.3 Refresh

The protected refresh route validates the same state/actor/tenant, cancels the old attempt
and creates a new hashed state and Account Link. It redirects immediately and never renders
or logs the link. Replays and cross-actor use fail safely.

## 7. Local Checkout Enablement

- OWNER may disable at any time;
- enabling first retrieves and synchronizes the provider account;
- enable succeeds only when the resulting connection remains READY and current;
- any synchronization loss of READY forces `enabledForCheckout=false`;
- every transition writes both append-only A4 Commerce audit and core `AuditLog` evidence;
- A6-B creates no Checkout Session, Order or Payment and does not make checkout public.

## 8. Settings Router And UI

Add a Commerce-owned protected router and `/settings/payments` Organisation tab.

Read model returns only bounded safe status, readiness booleans, safe counts/codes,
country/currency and timestamps. It returns no Account Link or state except from the
OWNER-only mutation that immediately initiates navigation.

UI states:

- not connected: Connect Stripe;
- created/onboarding: Continue setup;
- restricted: action-required explanation and Continue setup;
- READY disabled: Enable online payments;
- READY enabled: online payments enabled;
- DISABLED: provider-disabled explanation plus local disabled state;
- DISCONNECTED: retained-history explanation and Connect Stripe when no other current
  connection exists.

OWNER sees mutation controls. ADMIN sees status only. All states may link to the normal
Stripe Dashboard in a new tab. No secret/key/KYC/bank/card input exists.

## 9. URL And Environment Safety

- server URL base comes only from an explicit deployment URL (`NEXTAUTH_URL` or
  `NEXT_PUBLIC_APP_URL`), never a request-supplied host or mutation input;
- development may use `http://localhost`; non-development requires HTTPS;
- reject credentials, fragments, non-root base paths and unapproved schemes;
- return/refresh paths are fixed constants;
- credential mode is derived from `STRIPE_SECRET_KEY` (`sk_live_` versus non-live), never
  from tenant input or an Account payload;
- no new environment variable is needed until A6-D's separate Connect webhook secret.

## 10. Transaction, Failure And Audit Rules

- use SERIALIZABLE transactions and ordered tenant/connection advisory locks;
- no database transaction assumes a remote Stripe call can roll back;
- stable Stripe idempotency plus retained A4 operation evidence makes retries convergent;
- safe provider failures map to bounded application codes and redact provider detail;
- Account Link/state values never enter audit metadata, logs, URLs after return or stored
  response bodies;
- audit metadata may contain safe account ID, connection ID, resulting status, mode and
  boolean readiness only;
- subscription billing code and evidence remain untouched.

## 11. Validation

Required validation uses no real Stripe network call:

- pure mapping/controller/mode/URL/state validation;
- OWNER/ADMIN/MEMBER and cross-tenant authorization;
- new connection and existing connection resume with fake provider;
- stable local/provider idempotency and concurrent duplicate refusal;
- valid return, expired/replayed/cross-actor return and refresh;
- provider controller/account/mode mismatch and rollback/retry;
- all settings UI states and OWNER-only controls;
- enable only after fresh READY sync; disable at any time; readiness loss forces disable;
- A4 Commerce and core tenant audit evidence contains no secret/link/state;
- unchanged 140-migration inventory;
- A1-A6-A/C1-C6 and subscription-billing regressions;
- zero A6-B fixture residue.

## 12. Implementation Boundary

Expected files are limited to a Commerce Stripe Connect service/adapter and tests, one core
router registration, two protected return/refresh route handlers, the payment settings page
and component, settings navigation, and lifecycle documentation/roadmaps.

A6-B adds no Prisma model or migration, Connect webhook endpoint/secret, Checkout Session,
PaymentIntent, payment/refund mutation, Order, FUND behavior, email, account deletion,
Express login link or real Stripe test action.

## 13. Review And Acceptance Outcome

Review against the A6 parent, completed A6-A schema, pinned Stripe SDK types, current
protected tRPC/effective-tenant context and settings layout resolved:

- the pinned Accounts v1 controller fields express the accepted full Dashboard,
  account-fee, Stripe-loss and Stripe-requirements contract;
- `account_onboarding`, not `account_update`, is the valid hosted remediation link;
- full Dashboard accounts use the normal Stripe Dashboard and cannot use Express login
  links;
- no stable safe provider-side delete/disconnect operation is available for this contract,
  so local disable is implemented and provider disconnection remains event/sync evidence;
- server-owned deployment URLs and hashed single-use state prevent tenant-supplied redirect
  targets;
- injected provider boundaries permit complete testing without a real account or network;
- no schema change is needed and the accepted A6-A evidence supports the complete slice.

The plan is accepted for bounded A6-B implementation.

Implementation completed without changing this boundary. See the separate A6-B
implementation-confirmation and review/test records dated 2026-07-15.
